# v1.0-M1 — Self-Service Onboarding (Design / Spec)

**Date:** 2026-06-14
**Status:** Approved (brainstorm complete)
**Milestone:** v1.0-M1 (first of three milestones to the v1.0 acceptance bar)
**Parent bar:** `docs/superpowers/specs/2026-06-14-v1.0-acceptance-bar-design.md`

---

## Goal

Let a **tenant-admin self-serve everything within their own tenant** — mint
agent keys, register gateway upstreams (HTTP / OpenAPI), and supply per-tenant
upstream credentials — through a **console UI** backed by an **onboarding API**
(+ `runtimectl` parity), without an operator hand-editing `runtime.yaml` or
restarting the platform. Tenant *creation* remains superuser-only (the trust
boundary). This is "Gateway M4-lite" (dynamic, self-service upstreams + per-tenant
upstream credentials) fused with Identity's self-service tenant/key path.

## Design Decisions (settled in brainstorm)

| # | Decision | Rationale |
|---|---|---|
| 1 | **DB layer on top of file config** | File `gateway.servers` stay operator-owned platform defaults (load-once); a new `gateway_upstreams` Postgres table holds tenant-registered upstreams. Manager seeds from both; Add/Remove mutate only the DB layer. Lowest blast radius, clear ownership. |
| 2 | **Per-tenant *owned* connection + broker-injected credential** | A DB upstream is owned by ONE tenant, so its connection is not shared. The credential is a tenant secret (existing encrypted broker), injected into the upstream's headers at dial time. Sidesteps harness `tools/mcp` per-call-header limits. |
| 3 | **Extend server-rendered `html/template` console** | Matches the existing console pattern; no JS toolchain; right for a single-node self-host story. |
| 4 | **CSRF: synchronizer token (HMAC-of-session), keep `SameSite=Lax`** | First mutating console flow needs CSRF defense; stateless HMAC needs no new table. |
| 5 | **Two-tier RBAC** | Superuser creates the tenant + its admin; tenant-admin self-serves the rest within their tenant. Matches existing `requireAdmin`/`effectiveTenant`. |
| 6 | **Self-service transports: HTTP + OpenAPI only** | stdio `command:` = arbitrary host command execution; stays operator-only via file config. Structurally rejected in the DB table + API. |
| 7 | **Rotation-on-redial; no explicit reconnect action in M1** | Credential is just a broker secret; `secret rotate` / re-set works already; upstream picks up the new value on next (re)dial. |

## Architecture & Component Boundaries

Six units, each independently testable:

1. **`gateway_upstreams` table + gateway store** (new) — persists tenant-registered
   upstreams (HTTP/OpenAPI only), tenant-scoped, applied via the existing
   `store.ApplyDDLLocked` (advisory-lock + idempotent DDL) mechanism. Mirrors the
   identity/secrets schema style. The gateway package gains a store dependency.
2. **`gateway.Manager` mutability** (changed) — add `Add(cfg)` / `Remove(name)`;
   guard the upstream set with a lock (or atomic snapshot); per-upstream cancel
   funcs; bump `generation` on mutation so Handler caches rebuild (already wired).
   Seed from file config + DB rows at boot.
3. **Per-tenant credential injection** (new) — a dial wrapper pulls the named
   secret from the broker (`Broker.SecretsFor`) and merges it into the upstream's
   headers before connecting. Tenant-owned ⇒ not shared ⇒ no per-call header
   gymnastics. Fail-closed if the secret is missing.
4. **Onboarding API** (new, `controlplane/onboarding.go`) — upstream
   register/list/remove behind `requireAdmin` + `effectiveTenant`. Tenant +
   agent-key creation reuse the existing `/admin/tenants` (superuser) and
   `/admin/keys` (admin) routes; credential set reuses `/admin/secrets`.
5. **Console onboarding UI** (new, in `console`) — server-rendered pages/forms;
   first mutating console flow; adds a CSRF helper. **UI POST handlers and the
   `/admin/*` API call into the SAME store/manager/broker helpers** — one
   implementation, one set of validation rules.
6. **`runtimectl` parity** (new) — `runtimectl admin upstream add|ls|rm`,
   dispatched like the existing `register`/`secret` subcommands.

**Boundary check:** the store knows nothing about HTTP; the API layer knows
nothing about dial mechanics; the console calls the same surface a script would.

## Data Model & Credential Flow

### `gateway_upstreams` table

```sql
CREATE TABLE IF NOT EXISTS gateway_upstreams (
  id          text PRIMARY KEY,            -- generated, e.g. gwu-<rand>
  tenant_id   text NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
  name        text NOT NULL,               -- upstream name (no "__")
  transport   text NOT NULL,               -- 'http' | 'openapi' (never 'stdio')
  url         text,                        -- http transport
  openapi     text,                        -- openapi spec location (file/URL)
  base_url    text,                        -- openapi base override
  operations  text[],                      -- openapi operation filter (nil = all)
  cred_secret text,                        -- NAME of the tenant secret (nullable)
  cred_header text,                        -- header to inject into (nullable)
  created_at  timestamptz NOT NULL DEFAULT now(),
  UNIQUE (tenant_id, name)
);
```

- **Tenant-scoped & owned.** `tenant_id` = owning tenant (from the principal,
  never a client claim). `ON DELETE CASCADE` ties lifecycle to the tenant.
  Visibility maps to the Manager's `Tenants []string` as a single-element allowlist.
- **No credential material in this table.** `cred_secret` is only the *name* of a
  secret in the existing encrypted `secrets` table (sealed AES-256-GCM, AAD-bound
  to tenant+name). This table holds neither plaintext nor ciphertext — only a pointer.
- **stdio structurally impossible** — `transport` only ever `http`/`openapi`;
  enforced in API validation. stdio stays in file config.

### Credential flow (register → dial → call)

1. **Set credential** (UI/API) — tenant-admin sets a secret (e.g.
   `ORDERS_API_KEY=…`) via the existing `/admin/secrets` broker path → sealed.
2. **Register upstream** (UI/API) — names `cred_secret=ORDERS_API_KEY`,
   `cred_header=Authorization`. Row written; `Manager.Add` called.
3. **Dial** (gateway) — on connect, the dial wrapper calls
   `Broker.SecretsFor(ctx, tenant)`, looks up `cred_secret`, merges
   `cred_header: <value>` into the connection headers. Both self-service
   transports carry headers, so the credential routes through each transport's
   existing header mechanism: for **http** it merges into the Streamable-HTTP
   connection headers (`connect.go`); for **openapi** it merges into the
   generated tools' request headers (`restconn.go`). The wrapper is the single
   place that resolves the secret for both. Fail-closed: missing secret ⇒
   upstream `down` with a non-leaky error.
4. **Call** (agent) — unchanged: `mcp__gateway__<name>__<tool>`; the credential
   is already on the connection.

### Boot seeding & namespace

`Manager` is constructed from file `gateway.servers` **plus** all
`gateway_upstreams` rows. File and DB upstreams share one namespace; a name
collision is **rejected at registration time** (a DB upstream can never shadow a
file upstream).

### Rotation

Because the credential is just a broker secret, `secret rotate` / re-setting the
value works already; the upstream picks up the new value on its next (re)dial. No
explicit reconnect action in M1.

## API Surface & RBAC

### New routes (`controlplane/onboarding.go`, registered like `RegisterSecretAdmin`)

| Method / Route | Role | Tenant scope | Action |
|---|---|---|---|
| `POST /admin/upstreams` | admin | `effectiveTenant` (own; superuser names one) | Validate → write row → `Manager.Add` |
| `GET /admin/upstreams` | admin | own (superuser: all) | List upstreams + live status (`Manager.Status`) |
| `DELETE /admin/upstreams/{id}` | admin | own (no-oracle 204) | Remove row → `Manager.Remove` |

### Reused routes (unchanged)

- `POST /admin/tenants` — **superuser** (tenant creation = the trust boundary).
- `POST /admin/keys` — admin, own tenant (agent-key mint, one-time plaintext).
- `POST/GET/DELETE /admin/secrets` — admin, own tenant (credential set/list/remove).

### RBAC (two-tier)

- **Superuser**: creates the tenant + its first tenant-admin. Only cross-tenant authority.
- **Tenant-admin** (role=admin, scoped to `TenantID`): self-serves keys, upstreams,
  credentials within the tenant. `effectiveTenant` pins every write to the caller's
  tenant; the body's tenant field is ignored for non-superusers (existing behavior).

### Validation (at the API; reuse `config.Validate`-style checks)

- name unique within tenant + must not contain `"__"`.
- transport ∈ {http, openapi}; **stdio ⇒ 400**.
- exactly-one-transport; OpenAPI-only fields (`base_url`, `operations`) gated on
  `openapi`.
- `cred_secret` (if set) must reference an existing secret in the tenant;
  `cred_header` required iff `cred_secret` set.

### Errors / safety (project conventions)

Uniform 4xx with no token/secret oracle; **no secret value or secret name** in
logs or access log (only `tenant` / `upstream-id` / `transport`); broker or store
nil ⇒ 503; fail-closed everywhere.

### `runtimectl` parity

```
runtimectl admin upstream add --name <n> --url <u>
    [--openapi <spec>] [--base-url <b>] [--operations <csv>]
    [--cred-secret <s>] [--cred-header <h>]
runtimectl admin upstream ls
runtimectl admin upstream rm <id>
```

Dispatched exactly like the existing `register` / `secret` subcommands.

## Console Onboarding UI

Extends the `console` package — new `html/template` pages (auto-embedded),
`render()` helper, protected by `IdentityMiddleware`, principal from
`r.Context()`. Introduces the console's first mutating POST flows.

### Pages (under `/ui`, tenant-admin scoped)

1. **`GET /ui/onboarding`** — dashboard for the logged-in tenant-admin, one page,
   three sections:
   - **Agent keys** — list (id, role, label) + "mint key" form (role, label) →
     one-time plaintext shown once after POST.
   - **Upstreams** — list this tenant's upstreams with live status (up/down, tool
     count, last error from `Manager.Status`) + "register upstream" form (name;
     transport=http/openapi radio; url *or* openapi+base_url+operations; optional
     cred-secret dropdown + cred-header) + per-row remove button.
   - **Credentials** — list secret *names* (never values) + "set credential" form
     (name, value) → broker. Populates the cred-secret dropdown above.

2. **POST handlers** (same package): `POST /ui/onboarding/keys`,
   `/ui/onboarding/upstreams`, `/ui/onboarding/upstreams/{id}/delete`,
   `/ui/onboarding/secrets`. Each calls the **same** helper as the admin API,
   then POST-redirect-GET back to `/ui/onboarding` with a flash result.

### CSRF (synchronizer token, HMAC-of-session)

- Per-session CSRF token = HMAC(session token, server secret) — stateless, no new
  table. Embedded as a hidden `<input>` in every form via the template.
- Every POST handler verifies it (constant-time compare) before acting; mismatch
  ⇒ 403. Cookie stays `SameSite=Lax`. A small `csrf.go` helper owns issue + verify.

### UX details

- **Navigation**: an "Onboarding" link in the overview header, visible to admins
  only; routes 403 for viewer/operator.
- **One-time secrets**: minted keys / plaintext shown exactly once via a
  short-lived flash on the redirect page (not persisted, not re-displayable) —
  matching the CLI's "store this now" semantics.

## Testing

### Unit / hermetic

- **Manager mutation**: `Add` launches supervision + bumps generation; `Remove`
  cancels only that upstream's goroutine + closes its conn; concurrent
  Add/Remove/read race-free under `-race`; file-name collision rejected.
- **Credential injection**: dial wrapper merges `cred_header: <secret>` from a
  fake broker; missing secret ⇒ upstream `down`, non-leaky error; value/name
  never logged.
- **API handlers**: each `/admin/upstreams` route — RBAC (viewer 403, operator
  403, admin own-tenant 200, superuser cross-tenant with named tenant),
  `effectiveTenant` pinning, stdio→400, transport/field validation, no-oracle
  DELETE 204.
- **Console**: CSRF issue/verify (valid passes, tampered/absent → 403); each POST
  handler happy-path + redirect; onboarding page renders for admin, 403 for
  viewer/operator.

### Integration (`//go:build integration`, real Postgres, self-cleaning)

- `gateway_upstreams` store CRUD + tenant-cascade delete.
- End-to-end onboarding: superuser creates tenant + admin → tenant-admin mints
  key, sets credential, registers an HTTP upstream and an OpenAPI upstream →
  Manager picks them up (status up) → an agent call round-trips a tool through a
  registered upstream with the injected credential → cross-tenant access refused
  → remove upstream → gone from catalog.

### Live proof (M1 capstone — a strict subset of the eventual v1.0 stranger-install proof)

1. Start the platform (compose or local) with identity on, a superuser
   bootstrap, and the secrets keyring.
2. Superuser creates a tenant + tenant-admin (CLI or UI).
3. **In the browser**, as the tenant-admin: mint an agent key, set a credential
   secret, register a REST/OpenAPI upstream (the existing `examples/rest-demo`
   orders API) **and** a Streamable-HTTP MCP upstream — pointing the cred at the
   secret.
4. Drive an agent session: it discovers + calls a tool from the registered
   upstream, the injected credential authenticates the call, a cross-tenant
   attempt is refused.
5. Verify the access log shows `tenant` / `upstream-id` only — **no secret value
   or name**.
6. Remove an upstream in the UI; confirm it leaves the catalog.

**Discipline (every prior milestone):** holistic final review after all tasks,
then the live proof must pass clean before merge.

## Explicitly Out of Scope (deferred)

- stdio upstreams via self-service (operator-only, file config).
- Explicit "reconnect upstream now" action (rotation-on-redial is M1).
- OAuth2 upstream auth, rate limits/quotas, resources/prompts passthrough
  (Gateway v1.1).
- Open self-registration / SaaS-style tenant signup (superuser creates tenants).
- Console panels beyond onboarding (metrics, gateway-status).
- Per-tenant token/usage accounting (v1.0-bar defers to v1.1).
