# Runtime v1.0 — Acceptance Bar (Design / Spec)

**Date:** 2026-06-14
**Status:** Approved (brainstorm complete)
**Type:** Scope-definition spec (defines a release bar, not a single feature)

---

## The v1.0 Promise

> **Runtime v1.0** — A stranger can clone the repo, run one `docker compose up`
> on a single host, follow the docs to self-serve onboard a tenant **through the
> console UI**, and run real agents exercising all six AgentCore pillars
> (Runtime, Gateway, Memory, Identity, Sandboxes, Observability) — **without
> reading the Go source**. Proven by one clean from-scratch live run on a fresh
> machine, driven only by the published docs.

The platform's standing Goal is "an on-prem, open-source equivalent of AWS
Bedrock AgentCore." Every AgentCore pillar already has working, live-proven
milestones merged to `master`. **v1.0 is therefore not about new capability — it
is about turnkey self-hostability, self-service onboarding, and documentation,
proven by a stranger-install live run.**

## Scope Philosophy

Capability is **frozen** as of the milestones merged to `master` on 2026-06-13,
*plus* exactly the gap items enumerated in §"Per-Pillar v1.0 Cut". Everything
else is **Deferred to v1.1+** (enumerated explicitly below). The bar is a yes/no
Definition of Done, and the tag is gated on a single stranger-install
live-proof script. **No capability is added to v1.0 that is not required to make
the promise true.**

Three deliberate boundaries are baked in:

1. **Single-node Docker Compose is the v1.0 surface.** Kubernetes/Helm exists
   and keeps working, but is documented as "advanced" and is **not** part of the
   v1.0 promise or proof.
2. **Self-service onboarding is in; usage metering is out.** A tenant can
   onboard and add upstreams/credentials (via console UI and API), but
   per-tenant token/usage accounting is v1.1. This is a deliberate, eyes-open
   asymmetry: onboarding is a correctness/UX gate; metering is a billing nicety.
3. **Docs are a gate, not an afterthought.** "Without reading the source" means
   a quickstart, an operator guide, and a tenant guide must exist and be walked
   verbatim during the live proof.

## Per-Pillar v1.0 Cut (what ships in the tag)

| Pillar | In the v1.0 cut | Gap to close for v1.0 |
|---|---|---|
| **Runtime (spine)** | Durable multi-agent hosting, sessions, console, replica pools, autoscaling | None — done. Exercised by the proof. |
| **Identity** | Multi-tenant RBAC, OIDC + service keys, per-tenant secrets broker + rotation | **Self-service tenant onboarding** — tenant-create + agent-key mint callable without hand-editing `runtime.yaml`, surfaced in the console UI. |
| **Gateway** | MCP federation, semantic tool search, REST/OpenAPI adapters | **Self-service upstream registration + per-tenant upstream credentials** (Gateway M4-lite), wired to the secrets broker, surfaced in the console UI. |
| **Memory** | Durable store, semantic recall, auto-ingestion | **pgvector in the bundled Postgres** — today's bundled PG image lacks the extension, so the turnkey compose path silently lacks semantic Memory. **Hard requirement: semantic recall must work in compose out of the box.** |
| **Sandboxes** | Code interpreter + browser (egress proxy) | None functionally — included in the proof. (pip/kernel-persistence/per-user scoping stay v1.1.) |
| **Observability** | Prometheus metrics, Grafana dashboard, OTel tracing (Jaeger) | None — metrics + tracing is the agreed bar. (Token accounting + alerting → v1.1.) |

**Net new build for v1.0 = three things:**

1. **Self-service onboarding** — tenant + agent-key + upstream + per-tenant
   upstream-credential registration, via a **console UI** (primary) over an
   onboarding **API** (+ `runtimectl` parity commands). Spans Identity + Gateway.
2. **Turnkey single-node compose** — bundled **pgvector-capable** Postgres,
   one-command bring-up of all six pillars, fail-closed sane defaults,
   documented persistence/reset behavior.
3. **Docs runbook** — quickstart + operator guide + tenant guide — **plus** the
   stranger-install live-proof capstone.

## Explicitly Deferred to v1.1+

This list is load-bearing: it is what stops "just one more thing" from blocking
the tag. Each is a real capability the project will likely want, deliberately
left *out* of v1.0.

| Deferred item | Why safe to defer | Pillar |
|---|---|---|
| Per-tenant token/usage accounting | Onboarding is a correctness gate; metering is a billing nicety. The eyes-open asymmetry. | Observability |
| Alerting / recording rules | Metrics + Grafana let an operator *see* health; automated alerting is ops-maturity, not a self-hostability gate. | Observability |
| Kubernetes / Helm as a first-class path | Stays working, documented-as-advanced; v1.0 promise + proof are compose-only. | C2 |
| K8s operator / CRDs | The declarative production-deploy capstone — a post-v1.0 hardening milestone. | C2 |
| Sandboxes: pip-install, kernel-mode persistence, per-user scoping | Interpreter + browser work today; these are power-user ergonomics. | B4 |
| Memory: TTL/GC, dedup/merge, session synthesis, per-tenant embedding models | Core store + recall + auto-ingest are in; these are refinements. | B2 |
| Gateway: resources/prompts passthrough, OAuth2 upstream auth, rate limits/quotas | Tools federation + API-key/credential upstreams cover the v1.0 use. | B1 |
| Console panels **other than onboarding** (metrics panel, gateway-status panel) | The onboarding UI is in scope (see Per-Pillar Cut); richer panels are v1.1. | multiple |
| mTLS / per-agent non-shared registration tokens | Bearer-over-TLS is the v1.0 trust model; mTLS is hardening. | C3 |
| C1: PydanticAI adapter, follow-up-messages contract reconcile | Two foreign frameworks already host end-to-end; additive. | C1 |

Note: the **onboarding** console UI is explicitly **in** the v1.0 cut. Only
*other* console panels are deferred.

## Acceptance Gate — Stranger-Install Live Proof

v1.0 is tagged **only** when a single script + runbook passes clean, executed
**from scratch on a clean machine** (fresh clone; no pre-pulled images beyond
what compose declares; no author-only env). This is the same live-proof culture
applied to every prior milestone, scaled to the whole platform.

### Proof sequence

1. **Clone & up.** Fresh `git clone` → `docker compose up` (one command, per the
   quickstart doc verbatim). All services reach healthy; the bundled Postgres is
   pgvector-capable.
2. **Bootstrap operator.** Following the operator guide, create the first
   admin/operator (bootstrap path); log into the console.
3. **Self-serve onboard a tenant — through the console UI (browser):** create a
   tenant, mint its agent key, register an upstream (one **MCP** server *and*
   one **REST/OpenAPI** upstream), and supply a per-tenant upstream credential
   via the secrets broker. All steps performed in the browser.
4. **Run a real agent exercising all six pillars** for that tenant:
   - **Runtime** — agent boots; a session runs a multi-turn task.
   - **Gateway** — agent discovers + calls a federated tool (a semantic-search
     hit *and* a REST-adapter call).
   - **Memory** — a fact is written, then **semantically recalled** in a later
     turn (proves pgvector in compose).
   - **Identity** — the call is tenant-scoped; a cross-tenant attempt is refused.
   - **Sandboxes** — agent runs code in the interpreter and/or fetches a page
     via the browser tool.
   - **Observability** — the turn appears in metrics and as an OTel trace
     (Grafana + Jaeger reachable).
5. **Teardown.** `docker compose down` clean; documented data
   persistence/reset behavior holds.

### Gate rules

- **Docs-only:** driven solely by the published docs. If a step needs a command
  not in the runbook, the runbook is wrong — not the script. ("Without reading
  the source" is literally tested here.)
- **Fail-closed:** any pillar not demonstrably exercised ⇒ not v1.0.
- **In-repo + re-runnable:** the proof lives at `deploy/compose/v1-proof.sh`
  plus a runbook doc, so it is repeatable and CI-able later.
- **No secrets/secret-names in logs** (standing project rule) — the proof
  respects it.
- **The capstone is the forcing function:** M3's proof is also the integration
  test for M1+M2. A gap in onboarding or compose surfaces here and loops back.

## Build Sequence to Reach the Bar

Three milestones, dependency-ordered; each its own spec → plan →
subagent-execution → live-proof cycle. Then the capstone tag.

### v1.0-M1 — Self-service onboarding (Identity + Gateway + console UI)
The largest build. An onboarding **API** (tenant create, agent-key mint,
upstream register, per-tenant upstream credential set via the secrets broker),
`runtimectl` parity commands, **and a console onboarding UI** over it. This is
"Gateway M4-lite" (self-service upstreams + per-tenant creds) fused with
Identity's self-service tenant/key path, surfaced in the browser. Foundation:
the C3 M2 secrets broker already delivers per-tenant creds to agents — this
exposes the *registration* side.

### v1.0-M2 — Turnkey single-node compose
Make `docker compose up` honestly turnkey: swap the bundled Postgres for a
**pgvector-capable image** (hard requirement — unblocks Memory in compose),
one-command bring-up of all six pillars with fail-closed sane defaults,
documented persistence/reset. Loosely depends on M1 (onboarding must exist to be
wired in); cleaner after M1.

### v1.0-M3 — Docs runbook + stranger-install live proof (capstone)
Write the three docs (quickstart, operator guide, tenant guide), then build and
run `deploy/compose/v1-proof.sh` from a clean machine driven *only* by those
docs. Iterate docs until the proof passes clean. **When it passes, tag v1.0.**

**Distance to v1.0:** three milestones — all self-service / hardening / docs,
**zero new pillars**.

## Definition of Done (v1.0 tag checklist)

- [ ] **Onboarding:** a tenant can be created, an agent key minted, an MCP
      upstream and a REST/OpenAPI upstream registered, and a per-tenant upstream
      credential set — all via the **console UI**, backed by an API +
      `runtimectl` commands.
- [ ] **Compose turnkey:** a single `docker compose up` on a clean host brings
      up all six pillars healthy, with a **pgvector-capable** bundled Postgres.
- [ ] **Memory in compose:** semantic recall demonstrably works on the bundled
      compose deployment (not only with an external PG).
- [ ] **Docs:** quickstart + operator guide + tenant guide exist and are
      sufficient to perform the full proof without reading source.
- [ ] **Live proof:** `deploy/compose/v1-proof.sh` runs clean from a fresh
      clone on a clean machine, exercising all six pillars per the sequence
      above, driven only by the docs.
- [ ] **Hygiene:** no secrets/secret-names in logs or traces; `go build`,
      `go vet`, and the full test suite green.
- [ ] **Tag:** `v1.0` cut from the commit that passes the proof.
